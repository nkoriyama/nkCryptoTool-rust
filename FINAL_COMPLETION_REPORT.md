## 1. 概要

本レポートは、13サイクルにわたる継続的なセキュリティ強化および機能改善プロセスの集大成です。初期の PQC（耐量子計算機暗号）実装の不備や DoS 脆弱性、メモリ管理の課題を全て解決し、OpenSSL 3.5+ のネイティブ PQC サポートを含む、極めて堅牢で高性能なツールへと到達しました。

## 2. 主要な改善・修正内容

### 2.1 PQC 実装の完全化と高速化 (Report 56)
- **OpenSSL 3.5+ ネイティブサポート**: `EVP_PKEY_Q_keygen` 等の高レベル API を採用。外部プロバイダ不要で PQC をフルサポート。
- **圧倒的なパフォーマンス**: OpenSSL バックエンドでの PQC 暗号化速度が **~3.2 GiB/s** に到達（classical ECC と同等）。
- **クロスバックエンド相互運用**: RustCrypto と OpenSSL バックエンド間で鍵およびデータの完全な互換性を確保。
- **抽出ロジックの共通化**: `extract_raw_private_key` を `utils.rs` に集約。復号失敗時の「サイレント失敗」を構造的に解消。
- **ASN.1 構造的パースへの移行**: ヒューリスティックなバイト走査を廃止し、`pkcs8` および `spki` クレートを用いた厳格な構造検証を実装。

### 2.2 DoS 防御アーキテクチャの多層化 (Report 52-55)
- **真の Peer 識別 (PeerId)**: ハンドシェイクで確立された長期署名公開鍵の指紋に基づくクールダウン管理を実装。
- **ピア許可リスト (`--peer-allowlist`)**: 許可された鍵を持つユーザーのみ接続可能なモードを新設。
- **階層的タイムアウトとリソース制限**: `HANDSHAKE_TIMEOUT` (15s), `IDLE_TIMEOUT` (300s) 等の導入により、リソース占有攻撃を無効化。

### 2.3 メモリ機密性と安全性 (Report 51-56)
- **Lazy Loading 方式への移行**: 秘密鍵をメモリに常駐させず、必要な時だけロードし即座に破棄。メモリダンプ攻撃への耐性を最大化。
- **mlock によるメモリ保護**: ロードされた秘密鍵を `mlock(2)` で保護された領域に配置。

### 2.4 リソース最適化 (Report 54-55)
- **パイプライン署名・検証**: 署名処理をチャンク単位へ移行。巨大なファイルでも一定のメモリ消費で処理可能に。
- **ストリーム直接出力**: 受信データを一時ファイルを経由せず直接出力。平文露出リスクを排除。
