## 2. 実装詳細

### 2.1 PQC 鍵管理の正常化
- **形式の回帰**: 独自 SEQUENCE 形式を廃止しました。ML-KEM/ML-DSA の秘密鍵は、PKCS#8 の `privateKey` フィールドに生バイト列として格納される標準的な形式（多くの OpenSSL 実装等と互換）に戻しました。これにより F-42-1 (互換性) および F-42-2 (不要なシード露出) を解決しました。
- **厳格なバリデーション**: `unwrap_pqc_priv_from_pkcs8_inner` において、要求されたアルゴリズム（例: ML-DSA-65）に対応する期待サイズのみをヒューリスティック走査の対象とするように変更しました。これにより、ランダムデータ中に偶然現れるタグパターンによる誤検出を防止します。

### 2.2 全出力パスの保護
- **ハードリンク攻撃対策**: `sign_file`, `decrypt_file` に続き、`run_pipelined_encrypt` (暗号化出力) においても `create_new(true)` を導入しました。既存ファイルへの上書きや、ハードリンクによる他ユーザーファイルの破壊を未然に防ぎます。

### 2.3 ユーザー保護と可用性
- **チャットタイムアウト**: 24時間は単一スロットを占有するには長すぎるため、2時間 (CUMULATIVE_TIMEOUTと同等) に短縮しました。
- **高度な文字フィルタ**: Trojan Source 攻撃等で悪用される特殊な Unicode 文字（不可視文字や書字方向制御文字）のリストを大幅に拡充しました。

## 3. 検証結果

- **ビルド**: `cargo check` により構文および型チェックを通過。
- **整合性**: 鍵生成、暗号化、署名の全主要経路で `O_NOFOLLOW` + `O_EXCL` が一貫して適用されていることを確認。

## 4. 今後の検討事項

- **(F-42-1/2 関連)**: 将来的にシードを用いた鍵再生成が必要になった場合は、標準化団体 (NIST/IETF) の最終的な PKCS#8 策定内容に基づき、`SEQUENCE { seed, expandedKey }` 等の標準形式を構造的パーサを用いて実装すべきです。
