## 2. 実装詳細

### 2.1 実効的な DoS 対策 (F-46-1, F-46-2)
前回の実装では IP クールダウン判定が `CHAT_ACTIVE` フラグ取得後に行われていたため、攻撃者がフラグを一瞬だけ占有して解放するサイクルを繰り返すことで正規ユーザーを排除できる脆弱性がありました。
- **チェックの早期化**: 接続を受け入れた直後、かつグローバルな状態（フラグ）を変更する前にクールダウンを判定するように修正しました。
- **メモリ管理**: 新規接続ごとに 2 分以上経過した古いクールダウン記録を自動的に削除するようにし、メモリの無限増殖を防止しました。

### 2.2 非同期処理の最適化 (F-46-5)
ブロッキング Mutex を async コンテキストで使用する設計上の不整合を解消するため、`tokio::sync::Mutex` に移行しました。これにより、実行時のデッドロックリスクが低減され、他タスクのパニックによる巻き添え（poisoning）も発生しなくなりました。

### 2.3 安全なファイル上書きの実装 (F-46-9)
`--force` フラグ指定時でもセキュリティレベルを維持するため、既存ファイルを単純に `truncate` するのではなく、一度削除してから `create_new(true)` と `mode(0o600)` で再作成する手順を徹底しました。これにより、既存の不適切なパーミッション（0644等）の継承を完全に阻止します。

## 3. 検証結果

- **堅牢性検証**: `test_pqc_dsa_roundtrips` および `test_pqc_kem_roundtrips` を含むテストスイートを **20回連続で実行し、すべて PASS** することを確認しました。
- **DoS耐性**: 実装ロジックのトレースにより、複数の IP からの同時攻撃に対しても `CHAT_ACTIVE` フラグが保護されることを論理的に確認しました。
- **運用性**: `nk-crypto-tool --help` により `--force` フラグが表示され、意図通り機能することを確認しました。
