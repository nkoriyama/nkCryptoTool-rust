# Chat GUI セキュリティ・プロファイル (草案)

**作成日**: 2026-05-09
**状態**: 第57回分析サイクル向けインプット
**対象**: `nkCryptoTool-rust` Chat GUI (Slint 実装)

---

## 1. GUI 導入による新規脅威モデル

CLI から GUI への移行に伴い、信頼境界が「ターミナル・OSシェル」から「ウィンドウマネージャ・UIフレームワーク」へと拡大します。以下の新規脅威を検討します。

### 1.1 UI フレームワークによる機密漏洩 (Memory/Leak)
*   **脅威**: Slint の `LineEdit` や `Text` エディタ内部で、入力されたパスフレーズやメッセージが `Zeroize` されずにメモリに残存する。
*   **対策**:
    *   パスフレーズ入力には `input-type: password` を適用。
    *   フレームワークからの入力受取後、速やかに `Zeroizing<String>` へ転送。
    *   Slint 内部バッファのクリア方法を公式ドキュメントおよびソースコードで調査。

### 1.2 クリップボード経由の漏洩 (Clipboard)
*   **脅威**: Ticket や復号された機密テキストをクリップボードにコピーした際、他アプリやクリップボードマネージャに内容が盗み見られる。
*   **対策**:
    *   「クリップボードへコピー」機能に有効期限（例：30秒で自動消去）を設定。
    *   OS固有の「機密データ」フラグ（Android/iOS等）の利用。

### 1.3 画面表示の盗み見 (Visual/Capture)
*   **脅威**: スクリーンショット、画面共有、リモートデスクトップ、またはアクセシビリティ API 経由で、画面上の平文メッセージが読み取られる。
*   **対策**:
    *   ウィンドウ全体に対する「スクリーンショット防止」フラグの適用（OS サポートがある場合）。
    *   アクセシビリティ API からの機密フィールド除外。

### 1.4 通知センターへの露出 (Notification)
*   **脅威**: メッセージ受信時の OS 通知に、平文の内容や送信者が含まれ、ロック画面等で第三者に閲覧される。
*   **対策**:
    *   通知設定（デフォルト：内容非表示、送信者のみ表示など）のユーザー選択制。

## 1.5 M2: QR コードスキャン入力の脅威モデル

- **カメラ権限の最小化**: 常時 ON を禁止し、ユーザーが「Scan QR」ボタンを明示的に押下した期間のみデバイスをオープンする。
- **機密情報の非露出**: スキャンした QR ペイロードおよびカメラフレームをログ (stdout/stderr/file) に出力しない。
- **入力検証の強制**: `Ticket::from_str` によるパースに成功した文字列のみを UI 入力欄に注入する。

## 2. 継続される CLI セキュリティ要件

GUI 実装においても、以下の CLI 由来の防護策は「非妥協的要件」として継承されます。

1.  **Iroh V3.1 PQC ハンドシェイク**: プロトコル層の変更禁止。
2.  **コアダンプ無効化**: GUI プロセス全体で `setrlimit` を維持。
3.  **mlock による Swap 防止**: `SecureBuffer` の継続利用。
4.  **Lazy Loading**: 秘密鍵のライフタイム管理。

## 3. 分析が必要な領域

第57回以降のサイクルで以下の詳細分析を実施します。

*   **Slint Rust API のスレッド安全性**: UI スレッドと `chat_loop` スレッド間のデータ転送における競合状態。
*   **クロスプラットフォームでの `setrlimit` / `mlock` 等の挙動差異**: 特に Windows 環境下での機密保護の実効性。
*   **外部依存 UI ライブラリの監査**: Slint が依存するレンダリングエンジン（WGPU, Servo 等）のセキュリティ境界。
