## 実施内容 (再検証レポート v2 に基づく最終修正)

### 1. リソースクリーンアップの確実化 (F-IROH-27, F-IROH-25)
`EndpointGuard` による fire-and-forget な後処理を廃止し、`start()`, `run_connect()`, `listen_with_sender()` の各主要関数において、処理全体を `async` ブロックで包み、その終了後に確実に `endpoint.close().await` を待機するパターンに刷新しました。これにより、エラー発生時や早期 return 時も含め、Iroh のバックグラウンドタスクが確実に終了することを保証しました。

### 2. テスト範囲の正確な定義とスモークテストの追加 (F-IROH-30, F-IROH-33)
`test_iroh_chat_loop_e2e` を追加し、Iroh トランスポート上でハンドシェイクが完了し、チャットのメインループが正常に開始されることを確認しました。
- **制限事項**: 現状のテストは `stdin`/`stdout` に依存しているため、メッセージ内容の暗号・復号の自動検証は含んでおらず、起動と初期化を確認する「スモークテスト」として位置づけています。
- **今後の計画**: メッセージ内容の完全な自動検証（モックストリームの使用）は、構造的なリファクタリングが必要なため、フェーズ 3 以降の課題として継続します。

### 3. 未使用指紋フィールドの扱い (F-IROH-26)
Ticket フォーマットに含まれる `pqc_enc_fp` (ML-KEM 鍵指紋) について、現在のチャットプロトコル（ハンドシェイク V2）ではサーバ側が KEM 公開鍵を送信しないため、クライアント側での照合は実装されていません。
- **現状**: フィールド自体は将来の拡張性（フェーズ 3 のファイル転送等）のために維持し、現在は `sign_fp` による DSA 署名検証を MITM 防御の主軸としています。

### 4. フェーズ 3 へ継続する課題 (F-IROH-28, F-IROH-31)
以下の項目は、当初の計画通りフェーズ 3 のスコープで対応します。
- **F-IROH-28 (複数クライアント認証)**: クライアント側が自身の DSA 公開鍵を送信するプロトコル拡張を検討します。
- **F-IROH-31 (Relay 設定)**: `--no-relay` オプションおよび自前リレー指定の実装。
