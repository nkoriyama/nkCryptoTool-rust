## 1. 指摘事項への対応 (Ref: THREAT_ATTACK_SCENARIOS_v54.md)

| ID | 項目 | 内容 | 修正箇所 |
|:---|:---|:---|:---|
| **攻撃 A** | 複数鍵 DoS | `--peer-allowlist` を実装。許可された鍵のみ接続を許容し、攻撃コストを大幅に増大。 | `network.rs`, `main.rs` |
| **攻撃 B/C**| キャッシュ汚染 DoS | 脆弱性の原因となっていた `IP_TO_PEER_ID` キャッシュを完全削除し、設計を安全側に倒した。 | `network.rs` |
| **攻撃 D** | IP 汚染 DoS | ハンドシェイク失敗時の自動 IP クールダウン記録を廃止。共有 IP での妨害を排除。 | `network.rs` |
| **攻撃 E** | 1MB stall 攻撃 | `read_vec` の上限を 1MB から 8KB に縮小。メモリ浪費と接続ハングを防御。 | `network.rs` |
| **攻撃 F** | 片肺チャット | `rx_task` の終了を `tx` 側に通知するチャネルを実装。一方的な通信を防止。 | `network.rs` |
| **攻撃 G** | メモリダンプ攻撃 | キャッシュされた秘密鍵を `SecureBuffer` (`mlock`) で管理し、物理的な漏洩リスクを低減。 | `network.rs` |
| **攻撃 H** | 巨大ファイル OOM | `sign_file` および `verify_file` をパイプライン化。メモリ消費を一定（BUF_SIZE）に抑制。 | `processor.rs` |
| **攻撃 I** | 機能退行 (ECC) | PBES2 OID 検出ロジックを修正し、OpenSSL バックエンドでの暗号化 ECC 鍵の使用を復旧。 | `openssl_impl.rs` |
| **攻撃 M** | /tmp 平文漏洩 | ネットワーク経由の受信データを `NamedTempFile` を使わず直接 stdout へストリーム出力。 | `network.rs` |

## 2. 実装詳細

### 2.1 強固なアクセス制御 (攻撃 A)
同一攻撃者による「大量の使い捨て鍵」を用いたチャットスロット占有を防ぐため、許可リスト方式を導入しました。
- **Peer Allowlist**: 事前に信頼したピアの公開鍵指紋をファイルに記述し、`--peer-allowlist` で読み込むことができます。リストが有効な場合、未知の鍵による接続はハンドシェイク完了直後に拒否されます。

### 2.2 リソース消費の最小化 (攻撃 E, H, M)
DoS 耐性と大規模ファイル処理能力を同時に向上させました。
- **パイプライン署名**: これまでの「一括読み込み」を廃止し、暗号化と同様にチャンク単位でハッシュを更新するように修正しました。これにより、テラバイト級のファイルでも数メガバイトのメモリで署名可能です。
- **ネットワーク境界の厳格化**: ハンドシェイクで受信するベクトルのサイズを、実際の公開鍵サイズに即した 8KB に制限し、不当なメモリ確保を阻止します。

### 2.3 メモリ機密性の物理的強化 (攻撃 G)
- **mlock の活用**: 起動時にキャッシュされる署名秘密鍵を `mlock(2)` で保護された領域に配置し、OS によるスワップアウトや、意図しないメモリダンプへの露出を最小限に抑えました。

本サイクルにより、v54 で指摘された「必須対応項目」および「中期対応項目」のほぼすべてが完了しました。`nkCryptoTool-rust` は、単なる暗号ツールから、実運用上の様々な嫌がらせや攻撃に耐えうる「堅牢な通信プラットフォーム」へと進化しました。
