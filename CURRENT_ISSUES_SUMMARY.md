# nkCryptoTool-rust 現状の問題点まとめ

**日付**: 2026-05-05
**対象**: nkCryptoTool-rust (修正レポート 41–54 反映済み)
**目的**: v54 時点で**未解決のまま残っている問題**を全カテゴリで整理

---

## 概要

10 サイクル(v44 → v54)の修正を経て、暗号鍵管理と認証系のクリティカルな脆弱性は解決された。しかし以下のカテゴリで**実装上のバグ・設計上の根本課題・未対応の既存問題**が残存している。

| カテゴリ | 件数 | 重大度 |
|---|---|---|
| v54 で新規発生 | 4 件 | 🟠 重大〜中 |
| 既存攻撃シナリオ未対応 | 6 件 | 🟠 中〜🟡 軽微 |
| 設計レベル長期課題 | 3 件 | 🟠 中（先送り中） |
| 検証プロセスの問題 | 3 件 | 🟡 軽微 |

## 🔴 攻撃可能で重大な問題

### 1. 複数 long-term 鍵によるチャット占有 DoS（F-46-4 系、8 サイクル未対応）

**標的**: `network.rs:236-254`

**実害**:
- 攻撃者が ML-DSA-65 鍵を 100 個生成 (数秒)
- 各鍵で順次接続 → 切断 → 60秒 cooldown
- **100 × 60秒 = 1.7 時間のチャット完全占有**
- 鍵プールを大きくすれば永続 DoS

**根本原因**: 鍵生成コストが極めて低いため、PeerId::Pubkey ベースの cooldown を「異なる鍵」で回避可能。

**未対応理由**: F-46-4 から **8 サイクル連続**「今後の検討」と先送り。

**必要対策**:
- peer-pubkey 許可リスト（事前登録された鍵のみ受入）
- または PoW バインド（鍵生成コスト増加）
- または認証システム経由（PKI / Web of Trust）

---

### 2. IP_TO_PEER_ID キャッシュ汚染による選択的 DoS（v54 新規）

**標的**: `network.rs:38, 149-162, 246`

**実害**:
- VPN/CGNAT/Tor exit nodes 経由で同 IP の被害者を**60秒ブロック可能**
- 攻撃者は鍵を持っていれば handshake 完了 → 切断で cooldown 設定
- IP_TO_PEER_ID にキャッシュされ、同 IP の他ユーザーが弾かれる
- 60秒ごとに繰り返せば**永続 DoS**

**根本原因**: `IP_TO_PEER_ID` の早期チェックが「最後に接続した peer」の cooldown を同 IP の他者に適用してしまう（F-51-1 で解決した NAT 巻き添えが新形式で再発）。

**必要対策**: `IP_TO_PEER_ID` ベースの早期チェックを撤回。早期チェックは IP cooldown のみに留める。

---

### 3. IP_TO_PEER_ID 無限成長によるメモリ DoS（v54 新規）

**標的**: `network.rs:38`

**実害**:
- IPv6 SLAAC で大量 IP から接続 → HashMap 無限成長
- 1 接続/秒 × 1 年 = 約 2GB
- **数日〜数週間でメモリ枯渇**

**根本原因**: `IP_TO_PEER_ID` に `retain` による期限切れ削除がない。F-46-2 で対策された問題が**新キャッシュで再発**。

**必要対策**: 早期チェック内で:
```rust
let mut ip_map = IP_TO_PEER_ID.lock();
ip_map.retain(|_, peer_id| {
    cooldowns.get(peer_id).map(|t| t.elapsed() < Duration::from_secs(120)).unwrap_or(false)
});
```

---

### 4. handshake 失敗を悪用した IP cooldown 汚染（v54 新規）

**標的**: `network.rs:209-213`

**実害**:
- 攻撃者が VPN 出口 IP を選び、handshake で**意図的に署名を失敗**させる
- F-52-4 修正により失敗時に `PeerId::Ip(出口IP)` cooldown 記録
- 同 IP の被害者が早期チェック (2秒) で弾かれる
- 60秒ごとに繰り返し → 永続 DoS

**根本原因**: F-52-4 修正は handshake 失敗時のリトライ攻撃を防ぐ意図だったが、**選択的 DoS の経路を新設**。

**必要対策**:
- handshake 失敗時の IP cooldown を撤回
- または cooldown 期間を 5 秒以下に短縮

---

### 5. read_vec 1MB 上限による attack 増幅（潜在）

**標的**: `network.rs::read_vec`

**実害**:
- `read_vec` が長さプレフィクスを最大 1MB まで許容
- ECC 公開鍵 (91B), ML-KEM-1024 公開鍵 (1568B), ML-DSA-87 公開鍵 (2592B) と比較して**1000 倍過剰**
- 攻撃者が 1MB 長を要求して stall すると 100 並列で 100MB × 数秒の浪費

**必要対策**: `read_vec` の上限を 8KB 程度に縮小。

---

### 6. PBES2 OID 検出による ECC 暗号化 PEM の OpenSSL backend 機能退行（v54 新規）

**標的**: `openssl_impl.rs:715-722`

**実害**:
- v53 まで OpenSSL backend で動作していた **暗号化 ECC PEM** が v54 で起動不可
- エラーメッセージが「PQC keys not supported」となるため ECC ユーザーには意味不明
- v53 → v54 アップグレードの**運用退行**

**根本原因**: PBES2 OID は PQC 専用ではなく汎用の PKCS#5 v2 暗号化で使われる。OID 検出条件が誤っている。

**必要対策**: PBES2 OID 検出ブロックを撤廃するか、復号後の DER で PQC OID を再検証する設計に変更。

## 🟠 既存攻撃シナリオの未対応（v44 から残存）

### 7. 片肺チャット（v44 攻撃⑦）

`rx_task` がエラー終了しても tx 側ループは検知しない。被害者は「相手にメッセージが届いていない」ことに気付かないまま続報を送る → 情報漏洩リスク。

**対策**: `tokio::select!` で rx_task の終了を tx 側に通知。

### 8. cached_signing_key の長期メモリ常駐（v44 攻撃⑨ の悪化）

v44 までは数 ms の窓だった privkey メモリ滞在が、v49 の `preload_signing_key` で**プロセスライフタイム**に拡大。`gcore` や `/proc/<pid>/mem` でメモリダンプから抜き取り可能。

**対策**: 鍵を使う直前に展開・直後に zeroize する lazy ロード方式。

### 9. sign_file の OOM（v44 攻撃⑪）

`tokio::fs::read(input_path)` で全ファイルをメモリ展開。100GB ファイル署名で 100GB RAM 消費。`encrypt_file/decrypt_file` は pipelined 化されているが、**`sign_file` のみ memory-bound**。

**対策**: `sign_file` を pipelined 化（チャンク単位で sign_update / sign_final）。

### 10. /tmp 平文 leak（v44 攻撃⑩）

`network.rs::handle_server_connection` の非チャットモードで `NamedTempFile` に復号平文を一旦書く。/tmp 共有環境で他ユーザーに読まれる窓がある。

**対策**: ストリーム直結で `tokio::io::copy(&mut decrypted_stream, &mut stdout)`。

### 11. key_dir パストラバーサル（v44 攻撃⑧）

`config.key_dir` がユーザー指定で、`format!("{}/private_*.key", key_dir)` で任意パスに鍵書き出し可能。`secure_write` の `O_NOFOLLOW + 0o600` で安全だが、攻撃者が `key_dir` を制御できれば**意図しない場所**に鍵が配置される。

**対策**: `key_dir` を正規化、禁止プレフィックス（`/tmp`, `/var/tmp`, `..`）を拒否。

### 12. NAT 巻き添えの新形式（F-54-2 由来）

問題 2 と部分重複。既存の F-46-3 から 9 サイクルにわたって変動的に対応されている問題。
- v51 で IP cooldown 一律化により発生
- v52 で PeerId 導入により部分解決
- v54 で IP_TO_PEER_ID キャッシュにより**再導入**

## 🟡 設計レベル・先送り中の課題

### 13. ASN.1 構造的パーサへの未移行

**現状**: `unwrap_pqc_priv_from_pkcs8` 等で**ヒューリスティックなバイト走査**を継続。
- F-44-2 から複数サイクルで「中期目標」と位置付け
- 過去に複数の確率的バグの原因 (F-43-1, F-45-1)

**対策**: `pkcs8` クレートまたは `der` クレートへの移行。

### 14. グローバルレート制限の未実装

**現状**: 個別 cooldown のみ。**システム全体の秒間接続数制限**がない。
- F-46-4 から **8 サイクル連続「今後の検討」**
- ボットネット規模の攻撃に無防備

**対策**: トークンバケットアルゴリズムによるグローバル制限。

### 15. peer-pubkey 多重スロットの未実装

**現状**: チャットは単一スロット (`CHAT_ACTIVE`)。
- 同時に複数ユーザーがチャットできない
- 攻撃 1（複数鍵 DoS）が成立しやすい

**対策**: peer-pubkey ごとに独立したスロット管理。

## 🟡 検証プロセスの問題

### 16. OpenSSL backend テストの欠如

CI で `cargo test --no-default-features --features backend-openssl` を**走らせていない**。
- F-54-1 のような OpenSSL backend の機能退行を**永久に検出できない**

**対策**: CI で両バックエンドのテスト実行を必須化。

### 17. 統合テストのグローバル状態競合

`network::tests` の cooldown 関連テストが `CHAT_ACTIVE`, `PEER_COOLDOWNS` グローバル変数を直接操作。並列実行時に flaky になる可能性。
- F-49-6 で指摘済み、未対応

**対策**: `serial_test` クレートか `#[tokio::test(flavor = "current_thread")]` で直列化。

### 18. レポートと実装の不一致 (歴史的パターン)

**過去の典型例**:
- F-45 「全件 PASS」虚偽
- F-48-3 「F-47-3 緩和」虚偽
- F-49-1 復号せず暗号文を返す
- F-52-1 ephemeral 鍵を long-term identity として扱う

**v54 でも再発の可能性**: 「OpenSSL backend に PQC 鍵を投入した際、意図した通りの誘導メッセージ」と謳うが、これは**rustcrypto backend のテストでは検証できない**。手動確認のみで CI 自動化なし。

**対策**: 「修正完了」の主張前に**該当機能の自動テストが PASS していること**を必須化。

## 全体評価

### 達成された成果（v44 → v54）

- **暗号機能の根本動作**: F-43-1（ML-KEM 機能停止）、F-45-1（確率的署名失敗）、F-49-1（preload で復号せず）等の致命バグを解決
- **認証アーキテクチャ**: `allow_unauth=false` デフォルト化、PeerId::Pubkey 導入
- **DoS 防御の基礎**: cooldown、parking_lot::Mutex、HANDSHAKE_TIMEOUT
- **検証手段**: 統合テスト基盤、20回連続テストによる flaky 性確認

### 未解決の本質的課題

- **F-46-4 (複数鍵 DoS) が 8 サイクル連続未対応**
- **修正で新規問題を生むパターンの繰り返し**（F-49-1, F-50-1, F-52-1, F-54-1/2/3）
- **検証範囲の偏り**（rustcrypto backend のみ、グローバル状態テストの flaky 性）

### 結論

10 サイクルの集中的な修正で**実装上の致命バグ**は解決されたが、**設計レベルの根本対策**は依然先送り中。**現コードベースで本番運用するには即時対応必須項目（1〜5）の対応が必要**。

中期項目（6〜10）まで完了すれば、**個人〜小規模グループ用のチャット**としては実運用可能なレベルに到達する。

長期項目（11〜13）は、**公開サーバや大規模運用**を視野に入れた場合に必須。
