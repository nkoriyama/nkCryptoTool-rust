## 2. 実装詳細

### 2.1 コア機能の復旧 (F-43-1)
ML-KEM の復号において、PKCS#8 パーサが秘密鍵（DK）ではなく公開鍵（EK）のサイズを期待していたため、正常な秘密鍵が読み込めない致命的なバグを修正しました。FIPS 203 仕様に基づき、以下の正しい DK_LEN を設定しました：
- ML-KEM-512: 1632 bytes
- ML-KEM-768: 2400 bytes
- ML-KEM-1024: 3168 bytes

### 2.2 統合検証テストの導入
`cargo check` では検知できないロジック上の欠陥（サイズ不一致等）を早期に発見するため、以下の統合テストを `utils.rs` 内に実装しました：
- `test_ml_kem_768_roundtrip`: 鍵生成 → PKCS#8 包装 → 抽出 → カプセル化 → 復号の一連のフローを検証。
- `test_ml_dsa_65_roundtrip`: 鍵生成 → PKCS#8 包装 → 抽出 → 署名 → 検証の一連のフローを検証。

これらのテストは、`backend-rustcrypto` 機能を有効にした状態で正常にパスすることを確認済みです。

### 2.3 コードのクリーンアップ
- 未使用の `seed` 引数をインターフェースから完全に排除し、将来の誤用を防止しました。
- ネットワークパケットの境界値を再調整し、DoS 耐性を維持しつつ実用的な最小サイズを許容するようにしました。

## 3. 検証結果

- **単体テスト**: `cargo test --no-default-features --features backend-rustcrypto roundtrip` が **PASS**。
- **後方互換性 (F-43-8)**: v41 以前の標準的な PKCS#8 形式（OCTET STRING 形式）が正しく読み込めることをテストで間接的に確認しました。
