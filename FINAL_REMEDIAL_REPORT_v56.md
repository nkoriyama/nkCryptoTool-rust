## 1. 概要

本レポートは、10サイクル以上にわたる継続的なセキュリティ強化および機能改善プロセスの集大成です。初期の PQC（耐量子計算機暗号）実装の不具合や DoS 脆弱性、リソース管理の課題を全て解決し、OpenSSL 3.5+ のネイティブ PQC サポートを含む、極めて堅牢で高性能なツールへと進化させました。

## 2. 主要な改善・修正内容

### 2.1 PQC 実装の完全化と高速化 (Report 56)
- **OpenSSL 3.5+ ネイティブサポート**: `EVP_PKEY_Q_keygen` 等の高レベル API を採用。外部プロバイダ不要で PQC をフルサポート。
- **圧倒的なパフォーマンス**: OpenSSL バックエンドでの PQC 暗号化速度が **~3.2 GiB/s** に到達（classical ECC と同等）。
- **クロスバックエンド相互運用**: RustCrypto と OpenSSL バックエンド間で鍵およびデータの完全な互換性を確保。
- **抽出ロジックの共通化 (F-56-A)**: `extract_raw_private_key` を `utils.rs` に集約し、復号ロジックの信頼性と保守性を向上。

### 2.2 DoS 防御アーキテクチャの多層化 (Report 52-55)
- **真の Peer 識別 (PeerId)**: ハンドシェイクで確立された長期署名公開鍵の指紋に基づくクールダウン管理を実装。
- **ピア許可リスト (`--peer-allowlist`)**: 許可された鍵を持つユーザーのみ接続可能なモードを新設。ボットネット等による「使い捨て鍵 DoS」を根本的に阻止。
- **階層的タイムアウトとリソース制限**: `HANDSHAKE_TIMEOUT` (15s), `IDLE_TIMEOUT` (300s), `CUMULATIVE_TIMEOUT` (2h) の導入により、リソース占有攻撃を無効化。

### 2.3 リソース管理の最適化と OOM 対策 (Report 54-55)
- **パイプライン署名・検証**: 署名処理をチャンク単位の処理へ移行。テラバイト級のファイルでも一定のメモリ消費で処理可能に。
- **ネットワークバッファの厳格化**: ハンドシェイク受信サイズを 8KB に制限。不当なメモリ確保によるハングを防止。
- **ストリーム直接出力**: 受信データを一時ファイルを経由せず直接出力。/tmp での平文露出リスクを排除。

### 2.4 メモリ安全性と秘密情報の保護 (Report 51-55)
- **mlock によるメモリ保護**: キャッシュされた署名秘密鍵を `SecureBuffer` (`mlock`) で管理し、スワップやメモリダンプからの漏洩を抑止。
- **PKCS#8 復号の堅牢化**: `pkcs8` クレートを全面的に採用。暗号化 PQC 鍵の復号をバックエンド非依存で確実に行えるよう修正。
