# nkCryptoTool-rust 今後のロードマップ：未対応および拡張予定項目 (v56)

**日付**: 2026-05-05
**対象**: nkCryptoTool-rust v56 (コア脆弱性修正サイクル完了時点)
**ステータス**: 致命的な脆弱性は解消済み。本リストは将来の機能拡張・運用強化タスク。

---

## 1. 機能拡張・運用支援（短期〜中期）

### 1.1 peer_allowlist の Hot-Reload 機能
- **現状**: 許可リストを更新しても、サーバを再起動するまで反映されません。
- **目的**: 運用中の鍵失効や新規ピアの追加をダウンタイムなしで実現します。
- **対策**: `SIGHUP` シグナル送信による再読み込み、または `notify` クレート等を用いたファイル監視の実装。

### 1.2 許可リストファイルの属性保護 (F-55-4)
- **現状**: 許可リストファイル自体のパーミッション（権限）チェックが行われていません。
- **目的**: 悪意あるローカルユーザーによる許可リストの書き換えを防止します。
- **対策**: 起動時にファイルの所有権と `0o600` 等の厳格なパーミッションを強制チェックする機能。

### 1.3 key_dir パスの検証とバリデーション
- **現状**: `--key-dir` に指定されたパスの正規化が不十分です。
- **目的**: 意図しないディレクトリ（システムフォルダ等）への鍵ファイル出力を防止します。
- **対策**: パスの正規化（canonicalize）と、禁止ディレクトリ（`/tmp`, `/var` 等）への書き出し制限。

## 2. アーキテクチャ拡張（設計レベル・長期）

### 2.1 グローバルレート制限の導入 (F-46-4 系)
- **現状**: 個別 Peer に対するクールダウンはありますが、システム全体での「秒間接続数上限」がありません。
- **目的**: 大規模なボットネットによる接続試行自体の洪水からサーバを保護します。
- **対策**: トークンバケット方式等を用いた、グローバルなレートリミッターの実装。

### 2.2 チャットセッションの多重スロット化
- **現状**: チャットセッションはサーバ全体で「同時に1人（1スロット）」限定です。
- **目的**: 正当な複数のピアが並行してチャットを利用できるように利便性を向上させます。
- **対策**: `CHAT_ACTIVE` フラグを拡張し、最大 N 接続まで管理可能なスロット機構の実装。

### 2.3 TPM のネットワークモード対応
- **現状**: TPM 保護された鍵は、ローカル操作（署名・暗号化）では使えますが、ネットワークリスナー側での認証には未対応です。
- **目的**: サーバ秘密鍵を HSM/TPM などのハードウェアに封じ込めたまま、セキュアに運用可能にします。
- **対策**: TPM 経由の署名インターフェースをネットワークハンドシェイク・ループへ統合。

## 3. 開発・検証プロセスの改善

### 3.1 テストのグローバル状態からの完全隔離 (F-49-6 系)
- **現状**: 一部のテストが依然として静的なグローバル変数（`CHAT_ACTIVE` 等）を共有しており、並列実行時に干渉する可能性があります。
- **目的**: CI 環境におけるテストの完全な決定論的動作（Flakiness の根絶）を保証します。
- **対策**: テストごとに独立した環境を動的に生成・破棄するリファクタリング。
